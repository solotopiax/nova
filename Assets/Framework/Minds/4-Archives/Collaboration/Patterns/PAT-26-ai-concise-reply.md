# PAT-26：AI 回复精简铁律

## 适用场景（When）

- 用户与 AI（主会话 / 子 agent）的所有最终回复输出
- 用户已在 `CLAUDE.local.md` 或会话开头声明偏好「简洁直接、不废话、不加收尾总结」
- 任务类型：命令执行 / 规范确认 / 代码改动汇报 / skill 落地反馈
- 触发信号：AI 即将输出"我做了 ABC、流程是 123、下一步……"这类自我总结段时，应立刻收敛

## 核心做法（What & How）

**默认回复结构**：

| 场景 | 推荐结构 |
|---|---|
| 命令 / 规范类问答 | 一行结论 + 必要时 1–2 句解释 |
| 代码改动 | 一句话说明改动要点 + 关键文件路径 + 一行下一步（如有） |
| Skill 落地 | 一句话告知落地路径，不复述 skill 流程 |
| 决策建议 | 结论 + 主要权衡（≤2 句），不展开备选方案细节 |

**展开条件（满足任一才展开）**：
1. 用户明确问"为什么 / 详细说 / 解释一下 / 对比一下"
2. 决策非显然且会影响后续多步（必须把不显然处讲清楚）
3. 用户判断需要的关键风险 / 限制（漏掉会出错）

**反向硬约束**：
- 禁结尾"总结 / 我刚才做了什么"段
- 禁复述用户的输入
- 禁复述 skill 流程或工具规范
- 禁用 emoji / 装饰性分隔线扩充篇幅
- 禁三段铺垫后才给答案

## 为什么这么做（Why）

- 用户是 Nova 项目作者 + 技术负责人，深度参与所有决策，**信息密度比铺陈更重要**
- 长输出污染上下文窗口，多轮后压缩频繁，反而降低协作质量
- 自我总结段对用户是噪声——diff / 文件路径 / 落地位置才是可消费产物
- "废话默认输出"会让用户失去对 AI 输出的快速扫读习惯，进而养成跳读，重要信息反而被错过

## 反模式（Anti-patterns）

- **代码改动后 5 段总结**：写完 Edit 后输出"我已完成以下工作：1. 修改了 X；2. 添加了 Y；3. 更新了 Z"——用户能看 diff，复述等于浪费
- **Skill 执行后复述流程**：调用 `/nova-obs patterns-to-inbox` 后回复"流程如下：第一步……第二步……"——skill 自身已渲染流程，AI 再写一次纯冗余
- **简单问答的 3 段铺垫**：被问"这个文件在哪"，回复"好的，让我先理解一下你的需求……根据项目结构……综合考虑……答案是 X"——直接给路径即可
- **emoji + 装饰列表凑长度**：用 ✅ ⚠️ 🚀 等 emoji 把单行结论拆成花哨清单，视觉噪声且违反项目「除非用户要求否则不用 emoji」铁律
- **预防性免责声明**：在每条建议前加"如果我理解正确的话 / 仅供参考 / 你可以根据实际情况调整"——降低信息确定性，用户已具决策权

## 跨项目复用提示

- **可直接搬**：本 Pattern 是通用 AI 协作沟通规范，与技术栈无关，任何 Claude / Cursor / Copilot 长期协作场景均适用
- **适配点**：
  - 协作者画像变了要重判：面向新手的教学场景反而需要展开（用户偏好相反）
  - 团队协作回复（多人能看到）可适度增加上下文铺垫，单用户深度协作场景从严执行
  - 异步沟通（PR 评论 / 邮件）保留必要客套，本 Pattern 主要管同步会话

# PAT-129：死代码判定：链路追溯而非单点 grep

## 适用场景
- 有人提出「某文件/某方法可以废弃了吧」时
- 候选目标是框架入口、引擎回调、或零显式调用点的代码
- 需要在删除前确认是否会从源头掐断某条机制

## 核心做法
1. **先核查事实，不附和结论**：不凭印象判废，先把链路画出来。
2. **识别引擎魔术方法**：`OnApplicationPause/Focus/Quit` 等 MonoBehaviour 魔术方法由引擎自动回调，grep 不到调用点是正常现象，**不能据此判定死代码**。
3. **向下游追溯完整链路**：入口 → 接口契约 → 基类/实现 → 监听者接口。判断「能否废弃」取决于下游是否还有人依赖广播，而非入口本身被谁调用。
4. **区分死代码 vs 预留扩展点**：零实现的 Listener 接口若注释明确「预留给业务插件可选实现」，属于预留扩展点，按「预留接口不删」原则保留。
5. **要废就废整条链路**：若确认放弃整套机制，需同时删入口+广播方法+监听接口；单删入口会留下永远收不到回调的孤儿接口。

## 判定输出

- **保留**：链路仍成立，或属于引擎回调 / 预留扩展点。
- **整链删除**：入口、广播、监听接口和实现同时失效。
- **暂缓**：链路事实还没查清，不进入删除动作。

## 反模式
- ❌ 单凭 grep 无调用点就判死代码（漏掉引擎回调/反射/序列化入口）。
- ❌ 把预留扩展点（零实现接口）当冗余删除。
- ❌ 只删入口文件，留下下游孤儿契约与孤儿实现。

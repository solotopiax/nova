# PAT-04：「改什么读什么」分层上下文策略

## 适用场景

- 项目体量较大，源码与文档长期共存
- 需要控制阅读面和上下文成本
- 任务不是全模块重构，而是局部实现、局部修文档或局部排查

## 核心原则

**md 做导航，cs 做真相。**

也就是：

- 先用 `Docs` 判断模块入口、文件拆分、公开 API、状态机和常见误区
- 再去读与本次改动直接相关的 `.cs` 文件和代码片段
- 只有文档缺失、过期或明显存疑时，才扩大到完整源码

## 决策矩阵

| 场景 | 推荐策略 |
|---|---|
| 要修改某个较大的类或模块 | 先读对应 `Docs`，再按文档指向精读相关 `.cs` |
| 要修改很小的类型 | 直接读 `.cs`，改完再补 `Docs` |
| 只想调用某模块 | 优先读 `Docs` 的公开 API 和示例，不展开实现 |
| 文档缺失或已漂移 | 退回代码，以代码为准并顺手修文档 |
| 纯格式化、纯局部重命名、纯行为无变更小修复 | 直接改代码；若公开事实未变化，可不扩读整套文档 |

## 小类型直读判据

满足任一即可直接读 `.cs`：

- 根类及全部 partial 合计小于约 150 行
- 体量很小的枚举、接口、配置对象、数据容器
- 文档本身比源码更冗长

## 这样做的原因

- 文档比源码更适合做入口和导航。
- 源码才是当前实现真相。
- 先导航后精读，比“上来全读源码”更省上下文。
- 改完当场同步文档，比末尾集中补文档更不容易漏。

## 反模式

- 上来就把一个模块所有 partial 全读完
- 明明有现成文档入口却完全跳过
- 文档已失真却继续盲信旧文档
- 改完代码不回刷对应文档，制造下一轮阅读噪音
